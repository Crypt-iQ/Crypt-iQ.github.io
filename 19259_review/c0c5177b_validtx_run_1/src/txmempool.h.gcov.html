     388                 :            : /**
     389                 :            :  * CTxMemPool stores valid-according-to-the-current-best-chain transactions
     390                 :            :  * that may be included in the next block.
     391                 :            :  *
     392                 :            :  * Transactions are added when they are seen on the network (or created by the
     393                 :            :  * local node), but not all transactions seen are added to the pool. For
     394                 :            :  * example, the following new transactions will not be added to the mempool:
     395                 :            :  * - a transaction which doesn't meet the minimum fee requirements.
     396                 :            :  * - a new transaction that double-spends an input of a transaction already in
     397                 :            :  * the pool where the new transaction does not meet the Replace-By-Fee
     398                 :            :  * requirements as defined in BIP 125.
     399                 :            :  * - a non-standard transaction.
     400                 :            :  *
     401                 :            :  * CTxMemPool::mapTx, and CTxMemPoolEntry bookkeeping:
     402                 :            :  *
     403                 :            :  * mapTx is a boost::multi_index that sorts the mempool on 5 criteria:
     404                 :            :  * - transaction hash (txid)
     405                 :            :  * - witness-transaction hash (wtxid)
     406                 :            :  * - descendant feerate [we use max(feerate of tx, feerate of tx with all descendants)]
     407                 :            :  * - time in mempool
     408                 :            :  * - ancestor feerate [we use min(feerate of tx, feerate of tx with all unconfirmed ancestors)]
     409                 :            :  *
     410                 :            :  * Note: the term "descendant" refers to in-mempool transactions that depend on
     411                 :            :  * this one, while "ancestor" refers to in-mempool transactions that a given
     412                 :            :  * transaction depends on.
     413                 :            :  *
     414                 :            :  * In order for the feerate sort to remain correct, we must update transactions
     415                 :            :  * in the mempool when new descendants arrive.  To facilitate this, we track
     416                 :            :  * the set of in-mempool direct parents and direct children in mapLinks.  Within
     417                 :            :  * each CTxMemPoolEntry, we track the size and fees of all descendants.
     418                 :            :  *
     419                 :            :  * Usually when a new transaction is added to the mempool, it has no in-mempool
     420                 :            :  * children (because any such children would be an orphan).  So in
     421                 :            :  * addUnchecked(), we:
     422                 :            :  * - update a new entry's setMemPoolParents to include all in-mempool parents
     423                 :            :  * - update the new entry's direct parents to include the new tx as a child
     424                 :            :  * - update all ancestors of the transaction to include the new tx's size/fee
     425                 :            :  *
     426                 :            :  * When a transaction is removed from the mempool, we must:
     427                 :            :  * - update all in-mempool parents to not track the tx in setMemPoolChildren
     428                 :            :  * - update all ancestors to not include the tx's size/fees in descendant state
     429                 :            :  * - update all in-mempool children to not include it as a parent
     430                 :            :  *
     431                 :            :  * These happen in UpdateForRemoveFromMempool().  (Note that when removing a
     432                 :            :  * transaction along with its descendants, we must calculate that set of
     433                 :            :  * transactions to be removed before doing the removal, or else the mempool can
     434                 :            :  * be in an inconsistent state where it's impossible to walk the ancestors of
     435                 :            :  * a transaction.)
     436                 :            :  *
     437                 :            :  * In the event of a reorg, the assumption that a newly added tx has no
     438                 :            :  * in-mempool children is false.  In particular, the mempool is in an
     439                 :            :  * inconsistent state while new transactions are being added, because there may
     440                 :            :  * be descendant transactions of a tx coming from a disconnected block that are
     441                 :            :  * unreachable from just looking at transactions in the mempool (the linking
     442                 :            :  * transactions may also be in the disconnected block, waiting to be added).
     443                 :            :  * Because of this, there's not much benefit in trying to search for in-mempool
